## User Constraints (from CONTEXT.md)

### Locked Decisions

- **D-01:** Use `hashicorp/random` `random_password` to generate RDS master passwords at plan time. Generated value written to `aws_ssm_parameter` SecureString AND passed to `aws_db_instance.password`. Single source of truth, fully offline, never hardcoded. HMAC salt generated same way.
- **D-02:** `lifecycle { ignore_changes = [password] }` on each `aws_db_instance`; `lifecycle { ignore_changes = [value] }` on SSM parameter. Allows out-of-band rotation without Terraform drift.
- **D-03:** NO `data "aws_ssm_parameter"` reads ŌĆö that requires a live call and breaks `make plan-check`. Secrets generated by `random_password` in the same config.
- **D-04:** Full proxy resource set (`aws_db_proxy` + `aws_db_proxy_default_target_group` + `aws_db_proxy_target` + IAM role + Secrets Manager secret) gated behind `count = var.enable_rds_proxy ? 1 : 0` with `default = false`. Zero proxy resources in the default plan.
- **D-05:** SSM-only constraint intact at MVP. The Secrets Manager secret the RDS Proxy mandates materialises only when `enable_rds_proxy = true`.
- **D-06 (reinterpretation):** ROADMAP criteria #3/#6 are satisfied as "declared, fully implemented, wired behind `enable_rds_proxy`" ŌĆö proxy will NOT appear in the default plan. Proxy module `endpoint` output MUST still resolve when flag is off (guard with `try(resource[0].endpoint, null)`).
- **D-07:** Both RDS instances default to `db.t4g.small`, `gp3` storage ~20 GB with `max_allocated_storage` enabled. Overridable via tfvars.
- **D-08:** Both instances: `deletion_protection = true`, `skip_final_snapshot = false`, explicit `final_snapshot_identifier`.
- **D-09:** AWS-managed keys: SSM SecureStrings use default `alias/aws/ssm`; RDS instances set `storage_encrypted = true` with no `kms_key_id` (defaults to AWS-managed RDS key). No customer-managed CMK this phase.
- **D-10:** Uncomment `module "rds_tenant"`, `module "rds_proxy"`, `module "rds_control_plane"`, `module "ssm"` in `envs/prod/main.tf` (underscore labels). Uncomment matching outputs. SSM exports only names/ARNs, never values.
- **D-11:** Pre-declare all module variables before uncommenting calls. Undeclared arguments fail `terraform plan` with "unsupported argument".
- **D-12:** Offline `make plan-check` is the verification gate. All chosen approaches keep the plan fully offline.

### Claude's Discretion

- Postgres major version ŌĆö default to 16; planner's discretion. Set `engine_version` explicitly and pin parameter-group family.
- Control-plane RDS SG ingress ŌĆö default to 5432 from task SG id (provisioner runs in-cluster), mirroring the tenant RDS SG pattern.
- SSM parameter naming/paths ŌĆö e.g. `/${name_prefix}/rds/tenant/master-password`, `.../rds/control-plane/master-password`, `.../hmac-salt`.
- What "tokens" SecureStrings to scaffold ŌĆö scaffold HMAC salt + both RDS master credential params as the concrete set; add clearly-named placeholders only for known consumers.
- `count`/`for_each` vs single-resource style, resource ordering, parameter-group and subnet-group structure.

### Deferred Ideas (OUT OF SCOPE)

- Customer-managed KMS CMK for SSM SecureStrings and RDS storage ŌĆö deferred to security-hardening pass.
- RDS Proxy activation (`enable_rds_proxy = true`) ŌĆö flip at ~30 active tenants.
- Control-plane `prevent_destroy` lifecycle max protection.
- `db.t4g.micro` further downsizing or m-class upsizing.
- Read replicas, PgBouncer-style pooling beyond RDS Proxy, cross-region DR.

## Summary

Phase 3 implements four Terraform modules (`modules/ssm`, `modules/rds-tenant`, `modules/rds-proxy`, `modules/rds-control-plane`) and wires them into `envs/prod`. The hard constraint throughout is keeping the offline `make plan-check` gate green: the plan must be fully computable with dummy AWS credentials and no live API calls.

The secret-generation chain ŌĆö `random_password` ŌåÆ `aws_ssm_parameter` (SecureString) + `aws_db_instance.password` ŌĆö is the central pattern. It is entirely offline (no data sources, no STS lookups), produces a deterministic plan, and stores the actual secret value only in the encrypted S3 state and the SSM SecureString. Every resource that touches credentials must use `lifecycle { ignore_changes = [password] }` (RDS) or `lifecycle { ignore_changes = [value] }` (SSM) to allow out-of-band rotation without Terraform drift.

The RDS Proxy module is the most structurally complex: it requires five resource types (proxy, default target group, proxy target, IAM role + policy, Secrets Manager secret + version) all gated behind `count = var.enable_rds_proxy ? 1 : 0`. The module's `endpoint` output must use `try(aws_db_proxy.this[0].endpoint, null)` so `envs/prod` wiring compiles validly when the flag is off and the plan shows zero proxy resources. The single forced exception to the SSM-over-Secrets-Manager rule is this proxy IAM auth secret ŌĆö RDS Proxy cannot authenticate against SSM; it requires Secrets Manager. It only materialises when the flag is flipped.

**Primary recommendation:** Implement in dependency order ŌĆö (1) `modules/ssm` first (no external inputs beyond `name_prefix`), (2) `modules/rds-tenant` + `modules/rds-control-plane` in parallel (both consume `subnet_ids` and `task_security_group_id`), (3) `modules/rds-proxy` last (depends on the tenant RDS `identifier`). Wire all four into `envs/prod` together and run `make plan-check` once to confirm the plan is green.

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| RDS master password generation | Module layer (ssm + rds-tenant/cp) | ŌĆö | `random_password` lives in the same config as the resources it feeds; no cross-module data sourcing |
| Secret storage (credentials) | AWS SSM Parameter Store | AWS Secrets Manager (proxy only) | SSM is ~16-20x cheaper; Secrets Manager only materialises when RDS Proxy is activated (D-05) |

### Pattern 1: random_password ŌåÆ SSM SecureString ŌåÆ aws_db_instance

The central D-01 chain. All three resources live in the same module (or the `ssm` module passes the generated `random_password.result` to the rds modules via output):

```hcl
# Source: registry.terraform.io/providers/hashicorp/random/latest/docs/resources/password
# Source: registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/ssm_parameter
# Source: registry.terraform.io/providers/hashicorp/aws/latest/docs/resources/db_instance

resource "random_password" "tenant_rds" {
  length           = 32
  special          = true
  # Exclude / @ " and spaces ŌĆö characters that can break Postgres connection strings
  override_special = "!#$%&*()-_=+[]{}<>:?"
}

resource "aws_ssm_parameter" "tenant_rds_password" {
  name  = "/${var.name_prefix}/rds/tenant/master-password"
  type  = "SecureString"
  value = random_password.tenant_rds.result
  # key_id omitted ŌåÆ uses default alias/aws/ssm (AWS-managed, D-09)

  lifecycle {
    ignore_changes = [value] # Allow out-of-band rotation without Terraform drift (D-02)
  }
}

resource "aws_db_instance" "tenant" {
  # ...
  username = "odoo_master"
  password = random_password.tenant_rds.result

  lifecycle {
    ignore_changes = [password] # D-02
  }
}
```

**IMPORTANT ŌĆö SSM module boundary decision:** Because `random_password.result` must be passed both to `aws_ssm_parameter` AND to `aws_db_instance.password`, the cleanest structure is to generate the `random_password` resources inside `modules/ssm` and export the raw `result` values as sensitive outputs to the root, which then passes them as `master_password` variables to the RDS modules. This avoids duplicating `random_password` resources and keeps SSM as the single authoritative generator. The SSM module outputs the **names/ARNs** to the provisioner contract and the **raw values** (sensitive) to the root for RDS wiring only.

Alternatively, generate `random_password` inside each rds module and pass the result to the SSM module. This risks separate passwords drifting from the SSM record if lifecycle/count diverges. The SSM-centric approach is safer.

### Anti-Patterns to Avoid

- **`data "aws_ssm_parameter"` anywhere in Phase 3 modules:** This makes a live API call ŌĆö the plan fails with dummy credentials. (D-03)
- **`data "aws_availability_zones"`:** Same problem ŌĆö live call, breaks offline plan. Already established in Phase 1 ŌĆö pass AZs via `var.subnet_ids`.
- **`data "aws_caller_identity"` or `data "aws_region"`:** Live STS/metadata calls. Do not use.
- **Putting `random_password` in both SSM and RDS modules independently:** Creates two separate passwords with no guarantee they match. Centralise in `modules/ssm` and pass the `result` output to RDS modules.
- **Inline `ingress`/`egress` mixing with `aws_vpc_security_group_ingress_rule`:** Do not mix inline blocks with the standalone rule resources ŌĆö perpetual drift. Use only inline blocks (matching Phase 1 pattern) or only standalone resources, not both.
- **Omitting `vpc_id` from rds-tenant and rds-control-plane module variables:** `aws_security_group` requires `vpc_id`. Pre-declare before uncommenting the module call.
- **`module "rds-tenant"` with a hyphen label:** Must be `module "rds_tenant"` (underscore) ŌĆö the reference `module.rds_tenant.*` already appears in `envs/prod/outputs.tf`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cryptographically random passwords | `random_string` with manual seeding, or hardcoded values in tfvars | `random_password` (`hashicorp/random`) | `random_password` marks `result` as sensitive; tracks password in state so Terraform doesn't regenerate on every plan |
| RDS Proxy IAM auth credentials | SSM Parameter Store | `aws_secretsmanager_secret` | RDS Proxy's `auth` block only accepts `secret_arn` ŌĆö it cannot read SSM directly; this is the sole mandated Secrets Manager usage |
| Conditional output resolution when `count = 0` | `length(aws_db_proxy.this) > 0 ? aws_db_proxy.this[0].endpoint : null` | `try(aws_db_proxy.this[0].endpoint, null)` | `try()` is idiomatic, shorter, and does not evaluate the index expression at all when the list is empty ŌĆö avoids the "index out of range" plan error |
| PostgreSQL parameter group family name | Guessing or hardcoding | Set `family = "postgres16"` explicitly | AWS enforces family/engine_version alignment; mismatches cause apply errors |

**Key insight:** The `random_password` ŌåÆ SSM ŌåÆ RDS chain is the established community pattern for keeping credentials out of HCL while maintaining offline plan capability. Deviating from it (e.g., reading from SSM via data source, or using `manage_master_user_password`) re-introduces either live-call dependencies or Secrets Manager costs.

## Common Pitfalls

### Pitfall 1: Undeclared Variables Cause Plan Failure ("Unsupported Argument")

**What goes wrong:** Uncommenting the `module "rds_tenant"` call in `envs/prod/main.tf` before declaring `subnet_ids`, `task_security_group_id`, `master_password`, and `vpc_id` in `modules/rds-tenant/variables.tf` produces: `Error: Unsupported argument. An argument named "subnet_ids" is not expected here.`

**Why it happens:** Terraform validates that every argument in a module call has a corresponding variable declaration before planning.

**How to avoid:** Pre-declare ALL variables (including `vpc_id`) in each module's `variables.tf` before writing the resource body or uncommenting the call. The CONCERNS.md already flags `subnet_ids`/`vpc_id` for rds-tenant as a known gap.

**Warning signs:** `terraform validate` reports "Unsupported argument" for any of the new module inputs.

### Pitfall 2: SSM Output Exposes Secret Value to State-Diff Readers

**What goes wrong:** Defining `output "tenant_rds_password" { value = module.ssm.tenant_rds_password_value }` in `envs/prod/outputs.tf` writes the raw password into the state file under a human-readable output key.

**Why it happens:** Terraform stores all output values in state; anyone with S3 read access sees the password.

**How to avoid:** `modules/ssm/outputs.tf` exports only `parameter_name` (the SSM path string) and `parameter_arn` ŌĆö never `aws_ssm_parameter.*.value`. The `random_password.result` is passed as a sensitive variable between modules at plan time; it lives in encrypted state but is never in a named output accessible to the provisioner.

**Warning signs:** `envs/prod/outputs.tf` or any module `outputs.tf` references `.value` on an `aws_ssm_parameter` resource.

### Pitfall 3: gp3 IOPS Argument Confusion

**What goes wrong:** Some community sources suggest `gp3` requires an explicit `iops` argument. On small instances (`allocated_storage < 400 GB`), setting `iops` explicitly may conflict.

**Why it happens:** `gp3` has a baseline of 3000 IOPS included at no extra cost for storage < 400 GB. The Terraform AWS provider applies the default automatically when `iops` is not set.

**How to avoid:** Omit `iops` for the MVP 20 GB storage size. The provider will plan with `iops = 3000` (the gp3 default). Only set `iops` explicitly if tuning beyond the free baseline. [ASSUMED: based on community issue reports; verify during plan-check output]

**Warning signs:** Plan-check error referencing "iops must be specified for gp3" ŌĆö if seen, add `iops = 3000` explicitly.

### Pitfall 4: aws_db_proxy_default_target_group Import Behaviour

**What goes wrong:** Running `terraform plan` after adding `aws_db_proxy_default_target_group` when the proxy doesn't exist yet (flag off) causes confusion about whether the resource will be created.

**Why it happens:** The Terraform docs state: "Terraform does not _create_ or _destroy_ this resource, since it implicitly exists as part of an RDS DB Proxy." It is automatically imported on resource creation and produces no action on destroy. When `count = 0`, the resource does not appear in the plan at all ŌĆö correct behaviour.

**How to avoid:** Understand that with `count = 0` the default target group is also absent from the plan; this is expected. When `count = 1`, the resource will appear as "import" on first apply, not "create".

### Pitfall 5: SSM Parameter Value Drifts from RDS Password After Rotation

**What goes wrong:** After an out-of-band password rotation that updates the RDS master password (outside Terraform), the SSM parameter still holds the old value. `terraform plan` shows no drift because both have `ignore_changes`, but the values are now out of sync.

**Why it happens:** `ignore_changes = [password]` and `ignore_changes = [value]` prevent Terraform from detecting the divergence.

**How to avoid:** This is acceptable behaviour by design (D-02). Document operationally that password rotation must update BOTH the RDS master password AND the SSM SecureString atomically. The SSM parameter is the authoritative reference for the provisioner and other consumers.

### Pitfall 6: Module Label Hyphen vs. Underscore

**What goes wrong:** Writing `module "rds-tenant"` (hyphen) in `envs/prod/main.tf` when `envs/prod/outputs.tf` already references `module.rds_tenant.endpoint` (underscore). Terraform will report the output as unresolvable.

**Why it happens:** HCL allows hyphens in module labels, but the reference syntax `module.rds_tenant.*` uses underscores. The two are NOT interchangeable.

**How to avoid:** Always use the underscore label `module "rds_tenant"`. Matches the CONCERNS.md fragile-area note and the existing commented output references.

### Pitfall 7: VPC ID Not in rds-tenant Module Scope

**What goes wrong:** `aws_security_group` requires `vpc_id`. The current `modules/rds-tenant/variables.tf` only declares `name_prefix`. Adding `aws_security_group.rds_tenant` without first adding `variable "vpc_id"` causes a validate error.

**How to avoid:** Add `variable "vpc_id"` to `modules/rds-tenant/variables.tf` and `modules/rds-control-plane/variables.tf` (and `modules/rds-proxy/variables.tf`). Pass `module.networking.vpc_id` from `envs/prod/main.tf`.

### Pitfall 8: final_snapshot_identifier Must Be Unique

**What goes wrong:** If the RDS instance is destroyed and re-created with the same `final_snapshot_identifier`, AWS rejects the creation with a snapshot name conflict.

**How to avoid:** Use a sufficiently unique value, e.g. `"${var.name_prefix}-tenant-rds-final"`. This is stable and predictable. If the same name conflicts on re-creation, append a timestamp suffix via a `local` or just delete the old snapshot first.

## AWS Provider v6 Specific Notes

### What Did NOT Change for This Phase's Resources

Based on the v6 upgrade guide (GitHub issue #41101 and HashiCorp blog):

- **`aws_db_instance.password`:** The `password` argument is **still supported** in v6. `manage_master_user_password` (which routes to Secrets Manager) is an optional alternative, not a replacement. D-01 correctly uses the `password` argument. [MEDIUM confidence ŌĆö confirmed absence in v6 breaking change list]
- **`aws_ssm_parameter`:** The only v6 change mentioning SSM was `aws_ssm_association` losing the `instance_id` attribute. `aws_ssm_parameter` itself is unchanged. [MEDIUM confidence]
- **`aws_db_proxy`:** No breaking changes mentioned in v6 upgrade guide for RDS Proxy resources. [MEDIUM confidence]
- **Inline `ingress`/`egress` blocks in `aws_security_group`:** These are **not removed** in v6 ŌĆö they remain functional. The recommendation to use `aws_vpc_security_group_ingress_rule` is a best-practice guidance, not a breaking change. The existing Phase 1 networking module uses inline blocks and the plan is green; the same pattern applies here. [ASSUMED: based on issue tracker evidence that inline blocks generate functional plans in v6; no hard removal found]

### What DID Change (Relevant to Phase 3)

- **`aws_db_instance.character_set_name`:** v6 enforces that this attribute is only valid for newly created instances (not replicas/restores). Not used in this phase ŌĆö no impact.
- **Boolean strictness:** String `"1"` / `"0"` for booleans no longer accepted. Use `true`/`false`. Phase 3 resources should use proper boolean literals everywhere.
- **v6 general:** Default encryption defaults may have changed for some resources (e.g. Redshift). For RDS, `storage_encrypted = true` must still be set explicitly ŌĆö there is no automatic default encryption change for `aws_db_instance`.
